# Target .NET 10, not .NET 8

Report 2 §1 specified .NET 8 as the target platform. The dev machine this project was built on only has the .NET 8 and .NET 10 SDKs installed, with 10 as the ambient default — `dotnet new sln` on this machine defaults to the `.slnx` format, which the .NET 8 SDK's MSBuild cannot parse at all (`error MSB4068: The element <Solution> is unrecognized`), and a real transitive package conflict (`Microsoft.CodeAnalysis.Common` pulled at both 4.0.1 by `FastReport.OpenSource` and 4.5.0 by `Microsoft.EntityFrameworkCore.Design`) resolves silently under the .NET 10 SDK's NuGet resolver but hard-fails restore (`NU1107`) under .NET 8.

Decision: target `net10.0`/`net10.0-windows` across all projects, use a classic `.sln` (portable to any SDK, not just the reason above), and pin the toolchain explicitly via `global.json` (`10.0.300`, `rollForward: latestFeature`) so CI and any other machine builds with the same SDK this was developed and tested against. Report 2's ".NET 8" is superseded by this decision.
